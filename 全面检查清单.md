# AI辅助分类财务舞弊 — 作业全面检查清单

> 本文件供 AI 或人工逐项检查作业完成情况。每项标注 `[PASS]` / `[FAIL]` / `[WARN]`，并附简要说明。

## 一、交付物完整性检查

### 1.1 文件是否齐全

作业要求提交三项交付物，检查是否存在：

- [ ] **交付物一**：标注后的 Excel 文件（`STK_labeled_*.xlsx`）
- [ ] **交付物二**：工作报告（`report_*.pdf` 或 `report_*.docx`）
- [ ] **交付物三**：人工标注与模型对比表（`validation_*.xlsx`）

### 1.2 文件命名规范

检查每个文件是否严格遵循命名格式：

格式：`前缀_姓名_学号_G组号.扩展名`

- [ ] Excel 标注文件：`STK_labeled_{姓名}_{学号}_G{组号}.xlsx`
  - 示例：`STK_labeled_张三_2021310001_G03.xlsx`
  - 组号不足两位需补零（如 G03，不是 G3）
- [ ] 工作报告：`report_{姓名}_{学号}_G{组号}.pdf`（或 `.docx`）
- [ ] 验证对比表：`validation_{姓名}_{学号}_G{组号}.xlsx`

**检查要点**：
- 三个文件中的姓名、学号、组号是否一致
- 组号是否与 `作业分配.xlsx` 中的分配结果匹配
- 文件扩展名是否正确

## 二、Excel 标注结果检查（交付物一）

### 2.1 基本结构检查

- [ ] 文件是否在原始 `STK_Violation_Main.xlsx` 基础上扩充（保留全部原始列）
- [ ] 原始数据的行数是否正确（应与所负责组的数据行数一致）
- [ ] 是否新增了 6 个字段列，且列名完全一致：

| 列序 | 字段名 | 预期位置 |
|------|--------|---------|
| 24 | `ann_related` | 第24列 |
| 25 | `ann_year` | 第25列 |
| 26 | `ann_fin_flag` | 第26列 |
| 27 | `ann_fin_info` | 第27列 |
| 28 | `third_party_flag` | 第28列 |
| 29 | `third_party_list` | 第29列 |

### 2.2 数据类型检查

逐列检查数据类型是否符合规范：

- [ ] **ann_related**：值只能是 `0` 或 `1`（整数），无空值
- [ ] **ann_year**：当 `ann_related=1` 时为年份列表（如 `[2017, 2018]`）；当 `ann_related=0` 时为 `null`（字符串 "null" 或空）
- [ ] **ann_fin_flag**：当 `ann_related=1` 时为 `0` 或 `1`；当 `ann_related=0` 时为 `null`
- [ ] **ann_fin_info**：当 `ann_fin_flag=1` 时为 JSON 列表；否则为 `null`
  - JSON 格式示例：`[{"year": 2017, "elements": ["收入", "资产"]}]`
  - elements 只能包含六大要素：资产、负债、所有者权益、收入、费用、利润
- [ ] **third_party_flag**：当 `ann_related=1` 且 `ann_fin_flag=1` 时为 `0` 或 `1`；否则为 `null`
- [ ] **third_party_list**：当 `third_party_flag=1` 时为 JSON 列表；否则为 `null`
  - JSON 格式示例：`[{"name": "...", "type": "供应商", "role": "..."}]`
  - 每项必须包含 `name`、`type`、`role` 三个键

### 2.3 前提条件一致性检查（关键逻辑）

这是最容易出错的部分，需逐行验证条件依赖关系：

**规则 T1**：`ann_related` 对每行都必须有值（0 或 1）

**规则 T2**：若 `ann_related = 0`，则：
- `ann_year` 必须为 null
- `ann_fin_flag` 必须为 null
- `ann_fin_info` 必须为 null
- `third_party_flag` 必须为 null
- `third_party_list` 必须为 null

**规则 T3**：若 `ann_related = 1`，则：
- `ann_year` 必须有值（非空列表）
- `ann_fin_flag` 必须有值（0 或 1）

**规则 T4**：若 `ann_fin_flag = 0`（仅非财务信息），则：
- `ann_fin_info` 必须为 null
- `third_party_flag` 必须为 null
- `third_party_list` 必须为 null

**规则 T5**：若 `ann_fin_flag = 1`（影响财务信息），则：
- `ann_fin_info` 必须有值（非空 JSON 列表）
- `third_party_flag` 必须有值（0 或 1）

**规则 T6**：若 `third_party_flag = 0`，则：
- `third_party_list` 必须为 null

**规则 T7**：若 `third_party_flag = 1`，则：
- `third_party_list` 必须有值（非空 JSON 列表）

> **统计提示**：统计各前提条件违反的行数，若超过总行数的 1% 需标记为严重问题。

### 2.4 ann_year 字段专项检查

- [ ] 年份是否为合理范围（如 2000—2025），不存在未来年份或过早年份
- [ ] 年份格式是否为整数（如 `2017`，而非 `"2017"` 或 `"17"`）
- [ ] 同一行中是否有重复年份（如 `[2017, 2017]` 应去重）
- [ ] 年份是否按升序排列（建议但非强制）

### 2.5 ann_fin_info 字段专项检查

- [ ] JSON 格式是否可正确解析（无语法错误）
- [ ] 每个条目是否包含 `year` 和 `elements` 两个键
- [ ] `elements` 中的值是否严格来自六大要素：`资产`、`负债`、`所有者权益`、`收入`、`费用`、`利润`
- [ ] 是否存在使用具体科目名（如"应收账款"、"营业收入"）替代要素名的错误
- [ ] 同一条目内是否有重复要素
- [ ] year 值是否在 ann_year 列表范围内（即 ann_fin_info 中的年份必须是 ann_year 的子集）

### 2.6 third_party_list 字段专项检查

- [ ] JSON 格式是否可正确解析
- [ ] 每个条目是否包含 `name`、`type`、`role` 三个键
- [ ] `name` 是否为完整的法定名称（而非简称或代号，如"某公司"）
- [ ] `type` 是否来自建议类型列表：客户、供应商、银行/金融机构、券商/保荐机构、会计师事务所、评估机构、自然人、其他企业
- [ ] `role` 是否描述了具体的配合行为（不能留空或过于笼统）
- [ ] 第三方是否排除了不应计入的主体：
  - 上市公司自身及其全资/控股子公司
  - 上市公司的实际控制人
  - 仅作为正常业务对手出现的主体
  - 配合监管调查的主体（被动配合）

### 2.7 空行处理检查

原始数据中存在 Activity 为空的行（如第 28、1325、2060 行等），检查：
- [ ] Activity 为空的行，所有 6 个标注字段是否均填写为 null（字符串 "null"）
- [ ] 空行是否未导致后续行数据错位

## 三、标注逻辑深度检查（抽样验证）

> 从全量数据中抽取 10—20 条记录，人工阅读 Activity 原文，验证标注结果的正确性。

### 3.1 任务一（ann_related）判断规则验证

**应标注为 1（年报相关）的情形**：
- [ ] 年度报告中存在虚假记载
- [ ] 年度报告中存在重大遗漏
- [ ] 年度报告中存在误导性陈述
- [ ] 合并报表数据存在错误

**应标注为 0（非年报相关）的情形**：
- [ ] 仅影响季报、半年报，未影响年报
- [ ] 资金占用、违规担保（不涉及年报信息披露差异）
- [ ] 实控人身份隐瞒（不导致年报数据错误）
- [ ] 内幕交易、市场操纵类违规
- [ ] 窗口期交易、问询函回复、公司治理等非年报问题

**边界案例检查**：
- [ ] 处罚书同时提及半年报和年报问题时，是否正确标注为 1（年报相关）
- [ ] 处罚书仅提及"半年度报告"而未涉及年度报告时，是否正确标注为 0

### 3.2 任务二（ann_fin_flag / ann_fin_info）判断规则验证

**应标注 ann_fin_flag = 1（影响财务信息）的情形**：
- [ ] 虚构销售收入、虚增营业收入
- [ ] 虚增资产（应收账款、存货、固定资产等）
- [ ] 少计费用、不当资本化
- [ ] 虚减负债
- [ ] 利润操纵

**应标注 ann_fin_flag = 0（仅非财务信息）的情形**：
- [ ] 未披露关联方关系（但财务数据无误）
- [ ] 未披露重大诉讼、担保事项
- [ ] 未按规定披露持股变动

**会计要素映射检查**：
- [ ] 虚构销售 → 应包含 `收入`（而非具体科目"营业收入"）
- [ ] 虚增应收账款 → 应包含 `资产`
- [ ] 少计费用 → 应包含 `费用`
- [ ] 利润虚增 → 应包含 `利润`
- [ ] 虚增存货 → 应包含 `资产`
- [ ] 隐瞒借款 → 应包含 `负债`
- [ ] 未分配利润调整 → 应包含 `所有者权益`

### 3.3 任务三（third_party_flag / third_party_list）判断规则验证

**应标注 third_party_flag = 1（有第三方配合）的情形**：
- [ ] 供应商配合签订虚假采购合同
- [ ] 客户配合确认虚假销售
- [ ] 银行/金融机构配合伪造流水
- [ ] 中介机构（券商、会计师事务所）参与造假
- [ ] 自然人代持股份掩盖关联关系

**应标注 third_party_flag = 0（无第三方配合）的情形**：
- [ ] 造假行为完全由公司内部人员实施
- [ ] 第三方仅以正常业务对手出现
- [ ] 第三方仅配合监管调查（被动配合）

**类型推断检查**：
- [ ] 第三方类型是否根据 Activity 上下文推断（而非仅凭名称）
- [ ] 例如：描述中提到"供应商"配合 → type 应为"供应商"，即使公司名称中含"贸易"

## 四、验证对比表检查（交付物三）

### 4.1 基本结构

- [ ] 文件格式为 `.xlsx`
- [ ] 是否针对 5% 样本（使用 `random_state=42` 抽取）
- [ ] 样本量是否约为总数据的 5%（允许 ±1% 偏差）

### 4.2 必备列检查

验证文件是否包含以下所有列：

- [ ] 原始 Activity 文本（完整保留）
- [ ] 人工标注：`ann_related`、`ann_fin_flag`、`third_party_flag`
- [ ] 模型 A 预测：三个字段
- [ ] 模型 B 预测：三个字段
- [ ] 模型 C 预测（若使用三个模型）：三个字段
- [ ] 每个模型每个字段的对比分类：TP / FP / FN / TN

### 4.3 F1 Score 计算检查

- [ ] 是否分别计算了每个模型在每个字段上的 F1 Score
- [ ] F1 计算公式是否正确：`F1 = 2 * (Precision * Recall) / (Precision + Recall)`
- [ ] 任务一 F1 是否在全量 5% 样本上计算
- [ ] 任务二 F1 是否仅在 `ann_related = 1` 的子样本上计算
- [ ] 任务三 F1 是否仅在 `ann_related = 1` 且 `ann_fin_flag = 1` 的子样本上计算
- [ ] 若某子集样本量为零，F1 是否标注为 N/A 并说明原因
- [ ] 是否有平均 F1 汇总行
- [ ] 是否标注了最优模型（用于全量任务的模型）

### 4.4 模型信息检查

- [ ] 是否使用了 2—3 个国产大模型
- [ ] 是否列示了每个模型的具体版号信息（如 `qwen-max-2025-01-25`、`DeepSeek-V3` 等）
- [ ] 模型选择是否合理（建议范围：DeepSeek、Kimi、Qwen-Max、GLM、豆包等）

### 4.5 人工标注质量

- [ ] 人工标注是否覆盖所有字段（不仅是三个 flag，还应包括 ann_year、ann_fin_info、third_party_list）
- [ ] 人工标注与模型标注的对比是否仅针对三个 flag 字段（ann_related、ann_fin_flag、third_party_flag）
